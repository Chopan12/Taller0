1. Es un sistema que facilita el registro de cambios sobre uno o más archivos a lo largo del tiempo. Esto se hace con el fin de poder recuperar ciertas versiones que pudieran ser requeridas más adelante. 
[fuente](https://git-scm.com/book/es/v1/Empezando-Acerca-del-control-de-versiones)
2. El cliente replica completamente el repositorio en su dispositivo. De esta forma, si un servidor llegase a “morir”, cualquiera de los repositorios de los clientes podría ser utilizado para restaurar el servidor. 
[fuente](https://git-scm.com/book/es/v1/Empezando-Acerca-del-control-de-versiones)
3. Los repositorios remotos son versiones del proyecto que se encuentran en internet. Un repositorio local serían las versiones del proyecto que se encuentran alojadas en el computador personal. 
[fuente1](https://git-scm.com/book/es/v1/Fundamentos-de-Git-Trabajando-con-repositorios-remotos)
[fuente2](https://colaboratorio.net/atareao/developer/2017/git-y-github-trabajando-con-repositorios-locales/)
4. Todos los archivos, incluyendo versiones anteriores, se encuentran en un repositorio. Ahí, todos los cambios que han sido hechos en el repositorio anteriormente estarán siempre disponible. 
[fuente](https://www.thomas-krenn.com/en/wiki/Git_Basic_Terms)
5. Staging Area es una única carpeta, generalmente ubicada en el directorio de Git, que almacena la información acerca de lo que contendrá el próximo commit. 
[fuente](https://git-scm.com/book/en/v1/Getting-Started-Git-Basics)
6. Significa que se ha marcado un archivo modificado en su versión actual para que vaya en el próximo commit.
[fuente](https://git-scm.com/book/es/v1/Empezando-Fundamentos-de-Git)
7. Confirmado (o commited) significa que los datos están almacenados de manera segura en la base de datos local. 
[fuente](https://git-scm.com/book/es/v1/Empezando-Fundamentos-de-Git)
8. Commit es un comando de git que es utilizado para cambiar la cabecera. Cualquier cambio comprometido no afectará al repertorio remoto.
[fuente](https://www.hostinger.es/tutoriales/comandos-de-git)
9. Clone es un comando de git que sirve para, una vez elegido un repositorio, crear una copia de este.
[fuente](https://www.atlassian.com/git/tutorials/setting-up-a-repository/git-clone)
10. Pull es un comando de git que descarga y reemplaza el contenido del repositorio local con el contenido del repositorio remoto.
[fuente](https://www.atlassian.com/git/tutorials/syncing/git-pull)
11. Push es un comando de git que sube y reemplaza el contenido del repositorio remoto con el contenido del repositorio local.
[fuente](https://www.atlassian.com/git/tutorials/syncing/git-push)
12. Fetch es un comando de git que descarga los commits, archivos y referencias del repositorio remoto en el repositorio local. 
[fuente](https://www.atlassian.com/git/tutorials/syncing/git-fetch)
13. Merge es un comando de git que crea un hilo independiente del repositorio para ser trabajado de manera independiente del repositorio original.
[fuente](https://www.atlassian.com/git/tutorials/using-branches/git-merge)
14. Status es un comando de git que muestra el estado del trabajo del directorio y del Staging area. Permite observar que cambios han sido hechos y cuales no.
[fuente](https://www.atlassian.com/git/tutorials/inspecting-a-repository)
15. Log es un comando de git que permite revisar y leer el historial de todo lo que ha sucedido en el repositorio.
[fuente](https://mijingo.com/blog/understanding-git-log)
16. Checkout es un comando de git que permite moverse entre ramas o recuperar archivos trabajados en otras ramas.
[fuente](https://git-scm.com/docs/git-checkout)
17. Una rama es una manera que tiene git para moverse de manera simple entre los distintos commits que se han realizado en el repositorio.
[fuente](https://git-scm.com/book/en/v1/Git-Branching-What-a-Branch-Is)
18. Funciona para marcar una versión específica de la rama, para poder ser obtenidas cuando sean necesarias. 
[fuente](https://www.atlassian.com/git/tutorials/inspecting-a-repository/git-tag)
